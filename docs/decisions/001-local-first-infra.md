# ADR-001: インフラ構成をローカル優先で開始し、将来 VM に移行可能な設計にする

- **日付:** 2026-04-07
- **ステータス:** 採用

---

## 背景

スクレイピング（特に初回バッチ: 2016〜現在の約10万レース）は 4〜7 日の長時間タスクになる見込み。
また Phase 5 完成後は毎週の定期実行も必要になる。

実行環境として以下の選択肢があった：

| 選択肢 | メリット | デメリット |
|--------|---------|----------|
| ローカル Mac | セットアップ不要・コスト $0 | Mac の電源を入れっぱなしにする必要がある |
| クラウド VM（EC2/GCP 等） | 24時間安定稼働・Mac に依存しない | セットアップコスト・月 $10〜20 の費用 |

---

## 決定

**フェーズ初期はローカル Mac で実行する。**

理由：
- 中断・再開に対応した設計（取得済み `race_id` を DB で管理）になっているため、ネットワーク断・Mac シャットダウンのリスクは許容できる
- 初期フェーズはまず動くものを作ることを優先する
- 競馬開催日周辺（主に土日前後）のみ Mac をスリープしない運用で対応できる

---

## 将来の移行方針

以下の条件が揃ったタイミングで、クラウド VM への移行を検討する：

- 定期実行が安定し「常時稼働」の必要性を実感したとき
- Mac への負荷・電力コストが気になりはじめたとき

**移行コストは低い設計になっている：**

```bash
# .env の DATABASE_URL を変えるだけで接続先が変わる
DATABASE_URL=postgresql://user:pass@remote-vm:5432/inferenceeye
```

Docker Compose ごと VM に持っていくだけで移行できる。
データの移行は `pg_dump` → `pg_restore` で対応。

---

## 影響

- 初回バッチ実行中は Mac をシャットダウンしないこと
- 開催日の前日・当日は Mac をスリープしない設定にすること（Phase 5 実装後）
- 将来の VM 移行に備え、接続先は必ず環境変数（`DATABASE_URL`）経由で参照する
