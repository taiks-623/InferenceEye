# ADR-002: スクレイピング実行タイミングの方針

- **日付:** 2026-04-07
- **ステータス:** 採用

---

## 背景

Phase 2（スクレイピング実装）と Phase 3（モデル学習）の間には、
「コードが完成するタイミング」と「学習に使えるデータが揃うタイミング」にズレが生じる。

初回バッチ（2016〜現在の約10万レース）の取得に 4〜7 日かかる見込みであり、
これをいつ実行するか・開発フローのどこに組み込むかを明確にしておく必要があった。

---

## 決定

### 開発中の動作確認は「少量データ」で行う

各スクレイパーを実装するたびに、直近 1〜2 週間分のデータで動作確認する。
全件取得せずとも「DB に正しく入るか」「パースが正しいか」は少量で検証できる。

```bash
# 例: 直近 1 週間分だけ取得して動作確認
docker compose exec app python scraper/scrape_results.py --date-from 20260401 --date-to 20260406
```

### 初回バッチは Phase 2 完了後に実行する

全スクレイパーのコードが完成し、少量データでの動作確認が済んだ後に、
初回バッチ（2016〜現在）を実行する。

### Phase 3 はバッチ完了後に開始する

モデル学習には十分な量の過去データが必要なため、
初回バッチが完了してから Phase 3 に入る。

---

## 全体のタイムライン

```
Phase 2（コード実装）
  │
  ├── scrape_calendar.py 実装
  │     └── 少量データ（1週間分）で動作確認 ✅
  ├── scrape_results.py 実装
  │     └── 少量データで動作確認 ✅
  ├── scrape_shutuba.py 実装
  │     └── 少量データで動作確認 ✅
  ├── ... 他のスクレイパーも同様
  │
  └── Phase 2 完了
        ↓
  【初回バッチ実行】 ← 4〜7 日かかる待ち時間
  2016〜現在の全データを取得（Mac を動かし続ける）
  ※ この期間中にコードは変更しない
  ※ バッチ実行中に Phase 3 の設計を先に進めても良い
        ↓
  【初回バッチ完了】
  races テーブルに 10 万件以上のデータが入った状態
        ↓
  Phase 3 開始（特徴量・モデル実装）
```

---

## 経験則として残しておくこと

### コードの完成 ≠ データの準備完了

ML プロジェクトでは「コードが書けた」だけでは学習を開始できない。
データ収集・前処理のパイプラインが完成し、十分な量のデータが揃って初めて学習に入れる。

初回データ収集に数日かかることは珍しくなく、これをプロジェクト計画に織り込んでおくことが重要。

### 「少量 → 全量」の2段階で確認する

いきなり全件バッチを実行すると、数日後にパースのバグが発覚して最初からやり直しになるリスクがある。
少量データで動作確認を徹底してから全量を流す。

### バッチ実行中は設計・ドキュメント作業を進める

データ取得は待ち時間。その間に次のフェーズの設計を進めることで開発全体のスループットを上げられる。
