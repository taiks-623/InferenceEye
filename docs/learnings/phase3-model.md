# Phase 3 モデル設計 メモ

## ウォークフォワード検証の重要性

競馬データは時系列構造を持つため、ランダムな train/test 分割は使わない。
未来のデータで過去を学習するデータリークが発生するためである。

ウォークフォワード検証では学習期間を年単位で拡張しながら、常に「未来」を検証する：
```
Step1: 2016〜2019 で学習 → 2020 で検証
Step2: 2016〜2020 で学習 → 2021 で検証
...
```

---

## 評価指標は回収率が最重要

AUC（分離能力）は高いが回収率が低いモデルは意味がない。
競馬の本質的な目標は「オッズに対して期待値 > 1.0 の馬を見つけること」であり、
確率を正確に推定することよりも「人気（市場）が過小評価している馬」を見つけることが重要。

---

## 確率の正規化がゼロサムゲームに合わせる

LightGBM の raw 出力（sigmoid）はレース内の合計が 1 にならない。
同一レース内の馬の win_proba の合計を 1 に正規化することで「確率」として扱える。

```python
df["win_proba"] = df.groupby("race_id")["win_proba_raw"].transform(
    lambda x: x / x.sum()
)
```

---

## オッズのデータリークに注意

学習時: `results.win_odds`（確定オッズ）を特徴量に使う → 問題なし
推論時: `odds` テーブルの最新スナップショットを使う → 未来のデータではない

この非対称性を `predict.py` で適切に処理している。
確定オッズを学習に使うことは正当（「このオッズのとき誰が勝ったか」を学ぶ）。

---

## 直近データへの重み付け

競馬のトレンド（調教師の傾向、馬場改修、ルール変更等）は変化するため、
直近1年を 2.0 倍、直近2年を 1.5 倍の重みで学習する。
この値はハイパーパラメータとして Phase 4 で調整する余地がある。

---

## lightgbm/mlflow は関数内で import する

CI 環境（GitHub Actions）や開発環境によっては lightgbm の依存ライブラリ
（libomp等）が入っていない場合がある。
モジュールトップレベルで import すると、そのモジュールを import するテストが全滅する。
playwright と同じ lazy import パターンを使う。

```python
# NG: モジュールトップレベル
import lightgbm as lgb

# OK: 使用する関数の中
def _train_model(...):
    import lightgbm as lgb
    ...
```
